# ADR 005: 回归系统网页原型采用网页原生交互

## 背景
在将“游戏回归系统”转译为 HTML 页面时，存在一个明显风险：如果直接照搬手游界面的滑动轨道、强弹窗链路、场景式入口和高密度页签，网页会看起来像“浏览器里的伪手游”，既不符合网页用户习惯，也无法发挥网页在信息组织、扫读和跳转上的优势。

## 决策
1. **网页原型必须采用网页原生交互**：使用首屏摘要、固定导航、纵向任务列表、卡片、展开面板和轻量通知，而不是模拟手游中的左右滑动、轨道领奖或多层页签切屏。
2. **保留系统结构，不保留交互外壳**：欢迎激活、连续奖励、追赶任务、高价值福利、内容指引等结构继续保留，但以网页信息架构重新承载。
3. **将“游戏交互 vs 网页交互”写入页面正文**：原型不只展示视觉，更要明确说明为什么某些游戏交互在网页中被替换。
4. **HTML 原型进入 `exports/`**：作为规范的下游演示资产存在，服务验证和展示。

## 原因 (Why)
- **符合网页心智**：网页用户更依赖纵向浏览、固定定位和快速扫读。
- **降低认知摩擦**：避免让用户在网页里学习一套只适用于游戏的交互方式。
- **方便验证规范**：HTML 原型可以直接验证“页面合同式规范”是否足以支撑实际展示方案。

## 影响
- 后续把游戏系统转成网页原型时，应优先思考信息组织与交互改写，而不是视觉复刻。
- `exports/` 中新增 HTML 资产类型，用于承载规范的网页化演示。
