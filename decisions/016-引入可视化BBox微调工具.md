# ADR 016: 引入可视化 BBox 微调工具以补充 AI 识别管线

## 日期
2026-04-27

## 状态
已接受 (Accepted)

## 上下文 (Context)
在上游的 `NamingTool-AI` (Tauri App) 资产摄入管线中，我们使用了 VSOI 云端大模型（如 Gemini / GPT-4V）进行游戏 UI 的自动切图与坐标识别，并将结果以 JSON Manifests 的形式输出到本知识库的 `assets/star_rail_controls/metadata/screens/` 目录。
然而，AI 视觉模型对游戏 UI 的识别存在固有的像素级偏差（如裁剪过大、边界截断或将相邻组件误认为一体）。纯依赖重新调用 AI 成本高且结果不稳定，而手动修改 JSON 内的 `[x, y, width, height]` 坐标由于缺乏视觉反馈，操作极其繁琐痛苦。

## 决策 (Decision)
为了在工业级提取管线中加入一道人工阻断与微调阀门，我们在 `scripts/images/` 目录下引入了本地的 **可视化 BBox 微调工具 (`bbox_editor.py`)**。

1. **技术栈选型**：使用 Python 的 `Tkinter` 和 `Pillow` 构建轻量级桌面 GUI。无需额外前端环境，直接与现有的预处理脚本环境兼容。
2. **工作流嵌入**：
   - 上游系统生成包含 `bbox` 坐标的 JSON Manifest。
   - 使用 `bbox_editor.py` 直观地在一张包含原始游戏截图的画布上加载这些坐标。
   - 提供基于 Canvas 的 Hit-testing 交互（支持拖拽移动、拖拽四边及四角缩放）。
   - 修改结果通过快捷键直接序列化并覆盖回原始 JSON。
   - 随后继续走现有的 `extract_ui_crops.ps1` 脚本进行裁切和拼接输出。

## 后果 (Consequences)
**正面：**
- 将单张截图的坐标修正时间从数分钟缩短至 10 秒以内，实现了完全的“所见即所得”。
- 解耦了“AI 初筛”和“人工精修”环节，容忍了上游大模型生成坐标的不完美。

**负面：**
- 增加了一个需要手动执行的前端 Python 脚本，未完全达成 100% 自动化的闭环。但在当前多模态模型坐标能力尚未完全成熟的阶段，这是保证输出资产达到工业级标准的必要妥协。
