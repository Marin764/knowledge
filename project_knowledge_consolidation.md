## 记录 01：API 代理配置与标注工具优化 (Log 11)
**ID**: `11fe45f7-acdf-443c-859a-4ec40c615bcb` | **日期**: 2026-04-29

### 1. 核心目标
解决 `annotator.html` 在本地环境下（通过 CCSwitch 代理）调用 AI 识别接口时的 404 与 CORS 跨域问题，并探讨 UI 标注数据集的训练需求。

### 2. 关键对话与技术点
*   **API 配置调试**：
    *   **问题**：用户反馈 API 识别失败（404）。
    *   **诊断**：接口路径不完整（仅到端口）且存在浏览器 CORS 限制（由于 `file:///` 协议打开）。
    *   **方案**：补全 API 路径为 `http://127.0.0.1:15721/v1/chat/completions`，并建议使用本地服务器运行 HTML 以绕过跨域限制。
*   **数据集导出格式 (UI-UG DSL)**：
    *   导出文件：`custom_game_ui_dataset.json`。
    *   结构：采用 OpenAI 消息对格式，包含图片引用、页面 Prompt 以及助手生成的结构化 DSL（BBox、节点属性、交互关系）。
*   **微调 (Fine-tuning) vs 提示词工程 (Prompt Engineering)**：
    *   **提示词工程**：迭代快、成本低，但稳定性较差，受上下文长度限制。
    *   **微调**：稳定性高、响应快，能学习特定 DSL 风格。对于工业级需求，建议累积 500-1000 条高质量数据。
*   **标注工具功能增强建议**：
    *   探讨了“自动记忆用户操作”的可能性（例如自动为 Button 勾选 `onClick`）。目前的方案是使用 `Ctrl+C/V` 进行属性复制。

## 记录 02：工业化 UI 资产库与生图规范 (Log 10/2)
**ID**: `611c27e4-3e25-48be-ad5f-147db5b33fa7` | **日期**: 2026-04-27

### 1. 核心目标
建立基于 `assets` 库的工业级 UI 生成流，解决“规范输入 AI 后生成网页感界面”的偏科问题。

### 2. 关键决策与产出
*   **语义理解 vs 视觉模仿**：确立了利用带标注的元数据（BBox、Type）引导 AI 理解界面逻辑，而非单纯模仿素材。
*   **反网页化规则 (Anti-Web Rules)**：
    *   **空间槽位 (Spatial Slots)**：废除网页式的 `header/content` 描述，引入游戏专用的 `top_bar`, `left_rail`, `center_stage` (主舞台), `right_panel`。
    *   **构图职责**：强制定义区域的构图作用（如“沉浸式美术重心”），阻断流式布局。
*   **可视化 BBox 编辑器 (`bbox_editor.py`)**：
    *   **背景**：AI 自动识别的框体存在系统性偏移。
    *   **实现**：开发了基于 Python Tkinter 的可视化工具，支持通过拖拽、缩放实时修正 JSON 元数据。
*   **专题分析：回归系统 V3.0**：
    *   对《无期迷途》、《星穹铁道》、《王者荣耀》、《逆水寒》的回归系统进行了二次分析，应用最新的空间槽位模型。
*   **专题分析：剑与远征：启程**：
    *   **竖屏模型 (Portrait VSM)**：针对竖屏游戏建立了 `hero_stage`, `content_panel`, `bottom_bar` 等新槽位规范 (ADR 017)。
    *   完成了 137 张全量截图的数字化扫描与分类矩阵归档。

## 记录 03：命运曲线项目结构分析 (Log 3)
**ID**: `511ba1ab-ef7c-496e-a430-8bfbe18ea65e` | **日期**: 2026-04-27

### 1. 核心目标
分析并优化“基于命运曲线的历史人物生成交互系统 (Fate Curve)”原型项目。

### 2. 核心系统组成
*   **命运曲线 (Fate Curve)**：使用 ECharts 实现动态起伏。
*   **视觉雷达映射**：将命运强度实时转译为情绪张力、构图、色彩等 5 个视觉维度。
*   **长卷预览 (Digital Scroll)**：展示人物生平的视觉叙事。

### 3. 技术优化点
*   **多人物对比模式**：重构 `useCharacterStore` 以支持同时选择多个角色（如苏轼 vs 辛弃疾）。
*   **交互升华**：实现 ECharts 曲线的直接拖拽修改（基于 `graphic` 组件），并将折线优化为平滑贝塞尔曲线。
*   **美学反馈**：在长卷中引入动态滤镜（对比度、饱和度随强度变化）及云雾转场动画。

## 记录 08：数据存储架构升级 (Log 8)
**ID**: `05b60c5d-3a79-437a-86cc-2fbdd463856f` | **日期**: 2026-04-17

### 1. 核心目标
解决数据“动静分离”问题，保护协作项目的版本控制（SVN/Git）整洁性。

### 2. 存储策略划分
*   **协作数据 (`SnapNamingData/`)**：包含命名状态、UE 示意图、备注。这部分属于核心业务，**必须同步**。
*   **本地缓存 (`LocalCache/`)**：包含识图索引、临时切片。采用 MD5 哈希路径隔离，**不进版本控制**。
*   **应用数据**：AI 模型文件等，存放于应用安装目录。
*   **后端实现**：Rust 层引入 `get_app_storage_root` 仲裁机制，优先尝试便携模式，权限受限时回退至 `AppData`。
