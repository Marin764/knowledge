# Skill: 游戏系统交互规范生成器

> **触发方式**：当用户要求对某一游戏功能模块（如战令、商城、抽卡、公会、角色培养等）生成交互设计规范时，AI 遵循本 Skill 框架进行分析与输出。
>
> **输出目标**：生成一份正式的交互规范文档，保存至 `exports/[系统名称]-交互设计规范_V[N].md`。
>
> **输入要求**：用户需提供以下信息（或由 AI 从 `wiki/analysis/` 中推断）：
> - 目标系统名称（如：战令系统、商城系统）
> - 参考截图 or 已录入的 wiki 分析页面

---

**Role**: 你是一名拥有 5 年以上头部手游项目经验的 **UX 策划 Lead**，擅长将多款竞品的交互逻辑逆向工程，提炼为可被开发团队直接消费的规范文档。你熟悉米哈游、网易、叠纸、天美等顶级团队的界面设计范式。

**Core Principle**: 规范必须兼顾"框架通用性"与"系统特殊性"——先提炼跨产品可复用的设计模式，再标注系统级的差异化逻辑。

---

## 模块 1：核心信息架构 (Information Architecture)

**目标**：定义该系统 UI 在主视图中必须呈现的**数据字段**，并为每个字段设定展示优先级（P0/P1/P2）与可视化形式。

**输出格式**：

| 字段名 | 英文标识 | 展示优先级 | 可视化形式 | 来源案例 |
|------|------|------|------|------|
| [字段1] | `field.key` | P0/P1/P2 | 进度条/数字/图标列表 | 《游戏名》 |

**展示优先级定义**：
- **P0**：主视图常驻展示，用户进入页面即可感知，缺失会造成严重的操作障碍。
- **P1**：次要关键信息，需在操作前主动查看，缺失会影响决策效率。
- **P2**：辅助信息，可折叠或置于详情弹窗，缺失不影响核心流程。

**必须输出**：一个 Mermaid `graph TD` 信息架构树，展示主界面 → 各子模块的层级关系。

---

## 模块 2：状态机与交互流 (State Machine & Interaction Flow)

**目标**：定义系统中**核心交互元件**的状态枚举与合法的状态转换路径。

**通用状态机模板（AI 需按系统实际情况填充）**：

```
[初始状态 e.g. Locked / Empty / Inactive]
  → [触发条件 e.g. 条件满足 / 付费解锁 / 操作执行]
  → [等待确认状态 e.g. Available / Claimable / Selectable]
  → [用户操作 e.g. 点击 / 拖拽 / 确认]
  → [结束状态 e.g. Claimed / Used / Completed]
```

**AI 展开方向**：
1. **核心元件流**：识别该系统中最高频操作的元件（如奖励槽、商品格、角色卡），为其绘制完整状态机。
2. **主路径用户旅程**：使用 Mermaid `sequenceDiagram` 描述玩家从进入页面到完成核心操作的端到端交互链路。
3. **付费转化链（若该系统有商业化逻辑）**：必须描述从"发现付费价值"到"完成支付"的决策路径。

**强制要求**：状态机中必须包含**即时反馈 (Instant Feedback)** 节点——当用户完成关键操作后，系统应如何响应（动效、音效、数值刷新等）。

## 模块 3：布局与适配范式 (Layout Paradigms)

**目标**：基于竞品横评，提炼该系统界面的可选布局模式，并给出多端适配建议。

**AI 展开方向**：

### 3.1 布局模式对比（必须从 wiki 实例中归纳，≥ 2 种模式）

| 布局模式名称 | 视觉特征描述 | 代表产品 | 适用判断条件 |
|------|------|------|------|
| [模式 A] | [描述排列方式和视觉重心] | [产品名] | [什么样的系统适合用这种布局] |
| [模式 B] | ... | ... | ... |

### 3.2 响应式适配建议（必须针对主要设备给出规范）

| 设备类型 | 分辨率基准 | 布局调整策略 |
|------|------|------|
| 横屏手机（主场景） | 16:9 | [具体建议] |
| 竖屏手机 | 9:16 | [具体建议] |
| 平板 / 小屏 PC | 4:3+ | [具体建议] |

**安全区域强制规范**：所有可交互组件须在 `safeAreaInset` 范围内，避免与系统状态栏/Home Indicator 冲突。

## 模块 4：防坑与体验优化 (UX Best Practices)

**目标**：基于对竞品的 UX 诊断，总结该系统中已知的交互痛点与业内标准解法。

**输出格式（必须覆盖 ≥ 5 条）**：

| # | 痛点描述 | 典型错误场景 | 标准解法 | 参考来源 |
|---|------|------|------|------|
| 1 | [痛点] | [用户遇到的具体困境] | [设计上的解决方案] | [《游戏名》或通用规范] |

**AI 发掘方向**：
- **感知负担**：玩家是否需要记忆过多状态 / 点击路径是否超过 2 步？
- **反馈缺失**：操作完成后是否有即时的视觉/音效确认？
- **决策障碍**：付费或重要操作前，信息是否足够透明（如购买后立刻获得什么）？
- **高频操作效率**：最核心的操作是否提供了快捷路径（如一键操作）？
- **错误恢复**：误操作是否有撤回机制或确认弹窗？

## 模块 5：系统级抽象定义 (Component JSON Schema)

**目标**：提炼一个可被界面生成插件、低代码工具或原型系统直接消费的通用数据接口定义。

**AI 展开方向**：根据该系统的具体特征，填写以下模板：

```json
{
  "component": "[SystemName]Root",
  "version": "1.0",
  "system_type": "enum: battle_pass | shop | gacha | guild | cultivation | [其他]",
  "context": {
    "player_id": "string",
    "session_state": "enum: [根据系统枚举状态]"
  },
  "core_data": {
    "[系统核心数据块1]": { "注释": "根据系统填充字段" },
    "[系统核心数据块2]": { "注释": "如：物品列表、玩家资源、时间限制" }
  },
  "interactive_elements": [
    {
      "element_id": "string",
      "type": "enum: [此系统中的元件类型]",
      "state": "enum: [此系统的状态枚举]",
      "action": "enum: [可触发的操作]",
      "feedback": {
        "visual": "string (动效描述)",
        "audio": "string (音效描述)"
      }
    }
  ],
  "layout": {
    "paradigm": "enum: [从模块3中选择]",
    "primary_cta": "string",
    "entry_point": "enum: hub | standalone | modal"
  }
}
```

**注意**：模块 5 的 Schema 需体现该系统的特殊性，不应输出完全通用的空模板，而应基于模块 1-3 的分析结果填充系统专属字段。

---

## AI 执行规范 (Execution Protocol)

当触发本 Skill 时，AI 必须按以下顺序操作：

1. **确认系统范围**：明确用户要分析的系统名称，在 `wiki/analysis/` 中检索所有相关的 `type: ui_analysis` 分析页面作为素材来源。
2. **逐模块输出**：按模块 1 → 5 的顺序生成内容，每个模块的输出必须包含从真实案例中提炼的内容（不得凭空捏造）。
3. **生成文件**：将规范以 `exports/[系统名称]-交互设计规范_V[N].md` 格式保存，版本号递增。
4. **维护索引**：更新 `wiki/index.md` 中的 UX Spec 链接至最新版本。
5. **健康检查**：执行 `node scripts/lint.js` 确保全库 0 死链。
